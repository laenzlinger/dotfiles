# ADR 006: Use systemd-resolved for DNS resolution

**Date:** 2026-04-28

**Status:** Accepted

## Context

USB tethering (iPhone) intermittently fails to provide internet access despite NetworkManager successfully activating the connection and obtaining a DHCP lease. The browser cannot resolve any domains.

### Root cause

Netbird (VPN) and NetworkManager both write `/etc/resolv.conf` directly, creating a race condition:

1. NM writes `/etc/resolv.conf` with the correct upstream DNS (e.g. `172.20.10.1` from tethering)
2. Netbird overwrites it with `nameserver 100.89.175.51` (its internal DNS forwarder on the wireguard interface)
3. If the Netbird tunnel isn't fully established (common on new connections like tethering), its DNS forwarder is unreachable → all DNS resolution fails

The current state of `/etc/resolv.conf` is always Netbird's version:

```
# Generated by NetBird
search netbird.cloud
nameserver 100.89.175.51
```

NM has no `dns=` setting configured, so it defaults to rewriting `/etc/resolv.conf` — which Netbird immediately clobbers.

### Approaches considered

1. **Make `/etc/resolv.conf` immutable** — Prevents Netbird from overwriting, but also breaks Netbird's DNS for private network resources.

2. **Netbird `--disable-dns`** — Stops Netbird from managing DNS entirely. Loses the ability to resolve `*.netbird.cloud` names.

3. **systemd-resolved** — Acts as a local DNS stub (`127.0.0.53`). Both NM and Netbird configure their DNS through resolved's D-Bus API instead of fighting over a file. Resolved merges per-interface DNS servers and routes queries to the right upstream based on search domains.

## Decision

Enable `systemd-resolved` as the system DNS resolver.

### Changes required

1. Enable and start `systemd-resolved.service`
2. Symlink `/etc/resolv.conf` → `/run/systemd/resolve/stub-resolv.conf`
3. Set `dns=systemd-resolved` in `/etc/NetworkManager/conf.d/dns.conf`
4. Restart NetworkManager
5. Netbird auto-detects resolved and uses its API (no config change needed)

### Verification

- `resolvectl status` shows per-interface DNS servers (wifi, tethering, wireguard)
- USB tethering provides working DNS immediately
- Netbird `*.netbird.cloud` names still resolve when the tunnel is up
- Wifi DNS works as before

## Consequences

### Positive
- Eliminates the resolv.conf race condition between NM and Netbird
- Per-interface DNS routing: Netbird queries go to Netbird DNS, everything else to the upstream from wifi/tethering
- USB tethering works reliably without DNS breakage
- Standard Arch Linux approach (systemd-resolved is the preset default)

### Negative
- Adds a running daemon (`systemd-resolved`) — minimal resource impact
- DNS queries go through an extra local hop (stub resolver)
- If resolved crashes, all DNS fails (mitigated: it's a core systemd service)
